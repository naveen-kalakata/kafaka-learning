# Phase 2: Java 17 Through Real Application Code

## What this phase teaches

Java 17 is introduced only where it makes the code easier to read and safer to maintain.

## Language features used

- Records for DTOs and Kafka contracts
  Why: request bodies and event payloads are data carriers, not mutable domain objects.
- Switch expressions for task status decisions
  Why: status transitions are business rules, and switch expressions make those rules explicit.
- Text blocks in tests and docs
  Why: multi-line JSON and SQL are easier to read when escaped noise is removed.

## Why not use every Java 17 feature

- Sealed classes are useful, but they would add abstraction without improving this learning flow.
- Pattern matching is interesting, but it is not the best place to spend attention when JWT, Kafka, and Boot architecture are the real goals.

## Spring MVC to Spring Boot comparison

- Old Java 8 style often leans on mutable beans, long constructors, and utility classes.
- Java 17 lets us express intent directly with records and more readable control flow.
- The goal is not syntax novelty; it is lower accidental complexity.

## What to run

```powershell
./mvnw -pl shared-contracts test
```

## What should happen

- record-based DTOs serialize cleanly
- event contracts are small and obvious

## What failure looks like

- Java 8 compiler errors on `record`
- older IDE settings showing Java 17 syntax as invalid
